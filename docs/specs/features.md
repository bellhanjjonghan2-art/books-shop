# 기능 명세 — JWT 기반 로그인

> 본 문서는 `docs/login.md` (로그인 기능 PRD v1.0)를 기반으로, 기획 관점에서 "무엇을 만들지(요구사항)"를 사용자 스토리 단위로 재구성한 것이다.
> 구체적인 API 요청/응답 JSON 스키마 등 에이전트 간 인터페이스 계약은 본 문서의 범위가 아니다. **API 계약은 별도로 `docs/specs/api-contract.md`에서 프론트/백 개발자가 확정한다.** (참고로 `docs/login.md` 5절에 이미 구체적인 계약 초안이 작성되어 있으므로, 프론트/백 개발자는 해당 절을 그대로 검토·확정하면 된다.)

---

## 1. 범위 (Scope)

`docs/login.md` 1.1절 기준.

### 포함 (In Scope)
- 로그인 화면 (독립 컴포넌트)
- 로그인 API (`POST /api/auth/login`)
- JWT 발급 / 검증
- Spring Security 인증 필터
- Zustand 인증 상태 저장
- 인증 상태 기반 라우팅 가드

### 제외 (Out of Scope)
- 회원가입 기능 구현 (로그인 화면에 링크만 제공)
- 소셜 로그인 (OAuth)
- 비밀번호 재설정 / 찾기
- 이메일 인증
- Refresh Token 회전 (→ 9절 향후 확장 참고)

---

## 2. 사용자 스토리 및 수용 기준

### 스토리 1 — 로그인
**As a** 등록된 사용자
**I want to** 아이디(ID)와 비밀번호로 로그인하고 싶다
**So that** 인증된 사용자로서 서비스를 이용할 수 있다

**Acceptance Criteria**
- [ ] 올바른 ID/PW 입력 시 로그인에 성공하고, 서버가 정의한 응답(계약 문서 참고)을 받는다.
- [ ] 틀린 PW 또는 존재하지 않는 ID 입력 시, 두 경우 모두 동일한 401 오류 메시지를 받는다 (아이디 존재 여부를 노출하지 않음).
- [ ] LoginForm은 독립 컴포넌트로 분리되어 있다.
- [ ] LoginForm에는 userId, passwd(비밀번호 타입) 입력 필드와 회원가입 페이지로 가는 링크가 존재한다.
- [ ] userId 또는 passwd가 빈 값이면 클라이언트에서 제출을 차단하고 안내 메시지를 보여준다.
- [ ] 로그인 처리 중에는 로그인 버튼이 비활성화되고, 실패 시 에러 메시지 영역에 메시지가 표시된다.
- [ ] 비밀번호는 서버에서 bcrypt 해시로만 비교하며, 평문 비교는 일어나지 않는다.
- [ ] `/api/auth/login` 엔드포인트는 미인증 상태에서도 접근 가능하다.
- [ ] 서버 응답이나 로그 어디에도 사용자의 평문/해시 비밀번호(`passwd`)가 노출되지 않는다.
- [ ] 프론트엔드 origin에서의 로그인 요청이 CORS 설정으로 정상 처리된다.

### 스토리 2 — 인증 상태 유지
**As a** 로그인한 사용자
**I want to** 새로고침하거나 다시 접속해도 로그인 상태가 유지되길 원한다
**So that** 매번 다시 로그인하지 않아도 된다

**Acceptance Criteria**
- [ ] 로그인 성공 시 발급된 토큰과 사용자 정보가 Zustand 스토어에 저장되고, persist 미들웨어로 영속화된다.
- [ ] 새로고침 후에도 로그인 상태(`isAuthenticated`)와 사용자 정보가 유지된다.
- [ ] 인증이 필요한 요청에는 저장된 토큰이 `Authorization: Bearer` 헤더로 자동 첨부된다.
- [ ] 저장된 토큰으로 인증이 필요한 API(예: 현재 사용자 정보 조회)를 호출하면 정상적으로 사용자 정보를 받는다.

### 스토리 3 — 만료/위변조 토큰 처리
**As a** 시스템
**I want to** 만료되었거나 위변조된 토큰을 가진 요청을 거부하고 싶다
**So that** 인증되지 않은 접근을 차단하고 사용자를 안전하게 재로그인시킬 수 있다

**Acceptance Criteria**
- [ ] 만료되었거나 위변조된 토큰으로 인증이 필요한 API를 호출하면 401 오류를 받는다.
- [ ] 클라이언트가 401 응답을 받으면 인증 상태를 초기화(로그아웃 처리)하고 로그인 페이지로 이동시킨다.

> **확인 필요:** 위 401 처리가 모든 API 요청에 공통으로 적용되는 전역 규칙인지, 아니면 특정 보호 API에만 한정되는지 PRD 7.3절 표현이 다소 일반적이다. 프론트 개발 시 axios 인터셉터 적용 범위를 명확히 확인해야 한다.

### 스토리 4 — 라우팅 가드
**As a** 시스템
**I want to** 인증되지 않은 사용자가 보호된 페이지에 접근하면 로그인 페이지로 보내고 싶다
**So that** 인증이 필요한 화면이 보호된다

**Acceptance Criteria**
- [ ] 보호된 라우트에 `isAuthenticated`가 false인 상태로 접근하면 `/login`으로 리다이렉트된다.
- [ ] 로그인 성공 후, 원래 접근하려던 경로(또는 정해진 기본 페이지)로 이동한다.

> **확인 필요:** "로그인 성공 후 이동할 기본 페이지" 경로가 PRD에 구체적으로 명시되어 있지 않다 (예: `/`, `/books` 등). 또한 이 "원래 경로로 복귀" 동작이 PRD 8절 수용 기준 체크리스트에는 별도 항목으로 명시되어 있지 않아, 7.4절 본문과 8절 체크리스트 간 정합성을 확인해야 한다.

### 스토리 5 — 로그아웃
**As a** 로그인한 사용자
**I want to** 로그아웃할 수 있다
**So that** 내 인증 정보를 더 이상 노출하지 않을 수 있다

**Acceptance Criteria**
- [ ] 로그아웃 동작 시 Zustand 스토어의 토큰/사용자 상태가 초기화된다.
- [ ] 로그아웃 후에는 보호된 라우트 접근 시 다시 로그인 페이지로 리다이렉트된다.

> **확인 필요:** 로그아웃을 트리거하는 UI(예: 헤더의 로그아웃 버튼 위치/문구)가 PRD에 명시되어 있지 않다. 본 PRD 범위가 "로그인 기능"으로 한정되어 있어, 로그아웃 버튼/메뉴 UI는 별도 확인이 필요하다.

## 3. 도메인 및 데이터 요구사항

### 3.1 인증 흐름 (개요)
1. 사용자가 로그인 폼에 userId, passwd를 입력한다.
2. 클라이언트가 로그인 API를 호출한다.
3. 서버는 `users` 테이블에서 사용자를 조회하고, bcrypt로 비밀번호를 검증한다.
4. 검증에 성공하면 서버가 JWT를 발급한다.
5. 클라이언트는 발급받은 accessToken을 Zustand persist 스토어에 저장한다.
6. 이후 인증이 필요한 요청에는 `Authorization: Bearer <token>` 헤더가 첨부된다.

### 3.2 JWT 정책
| 항목 | 값 |
|---|---|
| 알고리즘 | HS256 |
| 토큰 타입 | Access Token (Bearer) |
| 만료 시간 | 1시간(3600초), 환경변수로 조정 가능 |
| 시크릿 키 | 환경변수 `JWT_SECRET`(256bit 이상), 코드 하드코딩 금지 |
| Claims | 사용자 ID(`sub`), 이름(`names`), 권한(`roles`), 발급/만료 시각(`iat`/`exp`) |

### 3.3 비밀번호 암호화 정책
- 저장 시 `BCryptPasswordEncoder`로 해시 후 저장 (strength 10 권장).
- 검증 시 입력값과 저장된 해시를 `matches()`로 비교하며, 평문 비교는 금지된다.
- 평문 비밀번호는 로그, 응답, DB 어디에도 남기지 않는다.

### 3.4 토큰 저장 방식과 트레이드오프
- 본 기능에서는 Access Token만 발급하며, 클라이언트는 이를 Zustand `persist`(localStorage)에 저장한다.
- 이는 단순성을 위한 선택이며, localStorage 저장은 XSS 위험에 노출될 수 있다는 트레이드오프가 있다.
- 보안 강화가 필요할 경우 Access Token은 메모리에만 두고 Refresh Token을 httpOnly 쿠키로 운용하는 방식으로 확장 가능하다 (→ 9절 향후 확장).

### 3.5 데이터 모델 — `users` 테이블
| 컬럼 | 타입 | 제약 | 비고 |
|---|---|---|---|
| user_id | varchar(100) | PK, not null | 로그인 ID |
| passwd | varchar(200) | not null | bcrypt 해시 저장, 평문 금지 |
| names | varchar(50) | not null | 사용자 이름 |
| birth | varchar(10) | not null | |
| gender | varchar(10) | not null | |
| phone | varchar(20) | not null | |
| email | varchar(100) | not null, unique | |
| address | varchar(300) | not null | |
| addr_detail | varchar(300) | default '' | |
| postcode | varchar(10) | not null | |
| roles | varchar(10) | not null | 예: USER, ADMIN |
| create_at | timestamp | default null | |
| update_at | timestamp | default null | |

로그인 기능에서 실제로 사용하는 필드는 `user_id`(로그인 ID 식별), `passwd`(인증 검증), `names`/`roles`(인증 응답 및 JWT 클레임 구성)이다.

> 본 기획 단계에서는 회원가입 기능이 범위 밖이므로, `users` 테이블에 실제 데이터를 채우는 절차(시드 데이터 등)는 본 문서에서 다루지 않는다.

## 4. API 계약 안내

로그인 API(`POST /api/auth/login`), 인증 확인 API(`GET /api/auth/me`), 공통 에러 응답 포맷 등 구체적인 요청/응답 스키마는 `docs/specs/api-contract.md`에서 프론트/백 개발자가 별도로 확정한다. (`docs/login.md` 5절에 초안이 이미 작성되어 있으므로, 개발자는 해당 절을 그대로 검토·확정하는 방식으로 진행할 수 있다.)

---

## 5. 향후 확장 (Backlog)

`docs/login.md` 9절 기준.

- Refresh Token 도입 및 토큰 회전 (httpOnly 쿠키 방식)
- 로그인 실패 횟수 제한 / 계정 잠금
- 회원가입 기능 PRD 연동
- `roles` 기반 권한 분기 (ADMIN 등)
- `update_at` 기록 (마지막 로그인 시각 등 운영 정책에 따라)

## 6. 확인 필요 항목 (PM → 팀장 보고 대상)

추측하지 않고 명확화가 필요해 보류한 항목이다.

1. **401 인터셉터 적용 범위**: 클라이언트의 401 응답 처리(로그아웃 + 리다이렉트)가 모든 API 요청에 전역적으로 적용되는지, 특정 보호 API에 한정되는지 PRD 7.3절 표현만으로는 확정하기 어렵다.
2. **로그인 성공 후 이동할 "기본 페이지" 경로**: PRD 7.4절에 "원래 가려던 경로 또는 기본 페이지로 이동"이라고만 되어 있고, 기본 페이지의 실제 경로(`/`, `/books` 등)가 명시되어 있지 않다.
3. **"원래 경로로 복귀" 동작의 수용 기준 정합성**: PRD 7.4절 본문에는 있지만 8절 acceptance criteria 체크리스트에는 해당 항목이 명시적으로 없다. 의도적 누락인지 확인이 필요하다.
4. **`GET /api/auth/me` 호출 시점**: "앱 진입 시 토큰 유효성 확인 용도"라고 되어 있는데, 이것이 앱 최초 로드(새로고침) 시 1회 호출인지, 매 페이지 진입마다 호출인지 명시되어 있지 않다.
5. **로그아웃 트리거 UI**: 로그아웃 버튼/메뉴의 위치나 문구가 PRD에 명시되어 있지 않다. 본 PRD 범위가 로그인 기능으로 한정되어 있어, 별도 확인이 필요하다.
