# 기능 명세 — JWT 기반 로그인

> 본 문서는 `docs/login.md` (로그인 기능 PRD v1.0)를 기반으로, 기획 관점에서 "무엇을 만들지(요구사항)"를 사용자 스토리 단위로 재구성한 것이다.
> 구체적인 API 요청/응답 JSON 스키마 등 에이전트 간 인터페이스 계약은 본 문서의 범위가 아니다. **API 계약은 별도로 `docs/specs/api-contract.md`에서 프론트/백 개발자가 확정한다.** (참고로 `docs/login.md` 5절에 이미 구체적인 계약 초안이 작성되어 있으므로, 프론트/백 개발자는 해당 절을 그대로 검토·확정하면 된다.)

---

## 1. 범위 (Scope)

`docs/login.md` 1.1절 기준.

### 포함 (In Scope)
- 로그인 화면 (독립 컴포넌트)
- 로그인 API (`POST /api/auth/login`)
- JWT 발급 / 검증
- Spring Security 인증 필터
- Zustand 인증 상태 저장
- 인증 상태 기반 라우팅 가드

### 제외 (Out of Scope)
- 회원가입 기능 구현 (로그인 화면에 링크만 제공)
- 소셜 로그인 (OAuth)
- 비밀번호 재설정 / 찾기
- 이메일 인증
- Refresh Token 회전 (→ 9절 향후 확장 참고)

---

## 2. 사용자 스토리 및 수용 기준

### 스토리 1 — 로그인
**As a** 등록된 사용자
**I want to** 아이디(ID)와 비밀번호로 로그인하고 싶다
**So that** 인증된 사용자로서 서비스를 이용할 수 있다

**Acceptance Criteria**
- [ ] 올바른 ID/PW 입력 시 로그인에 성공하고, 서버가 정의한 응답(계약 문서 참고)을 받는다.
- [ ] 틀린 PW 또는 존재하지 않는 ID 입력 시, 두 경우 모두 동일한 401 오류 메시지를 받는다 (아이디 존재 여부를 노출하지 않음).
- [ ] LoginForm은 독립 컴포넌트로 분리되어 있다.
- [ ] LoginForm에는 userId, passwd(비밀번호 타입) 입력 필드와 회원가입 페이지로 가는 링크가 존재한다.
- [ ] userId 또는 passwd가 빈 값이면 클라이언트에서 제출을 차단하고 안내 메시지를 보여준다.
- [ ] 로그인 처리 중에는 로그인 버튼이 비활성화되고, 실패 시 에러 메시지 영역에 메시지가 표시된다.
- [ ] 비밀번호는 서버에서 bcrypt 해시로만 비교하며, 평문 비교는 일어나지 않는다.
- [ ] `/api/auth/login` 엔드포인트는 미인증 상태에서도 접근 가능하다.
- [ ] 서버 응답이나 로그 어디에도 사용자의 평문/해시 비밀번호(`passwd`)가 노출되지 않는다.
- [ ] 프론트엔드 origin에서의 로그인 요청이 CORS 설정으로 정상 처리된다.

### 스토리 2 — 인증 상태 유지
**As a** 로그인한 사용자
**I want to** 새로고침하거나 다시 접속해도 로그인 상태가 유지되길 원한다
**So that** 매번 다시 로그인하지 않아도 된다

**Acceptance Criteria**
- [ ] 로그인 성공 시 발급된 토큰과 사용자 정보가 Zustand 스토어에 저장되고, persist 미들웨어로 영속화된다.
- [ ] 새로고침 후에도 로그인 상태(`isAuthenticated`)와 사용자 정보가 유지된다.
- [ ] 인증이 필요한 요청에는 저장된 토큰이 `Authorization: Bearer` 헤더로 자동 첨부된다.
- [ ] 저장된 토큰으로 인증이 필요한 API(예: 현재 사용자 정보 조회)를 호출하면 정상적으로 사용자 정보를 받는다.

### 스토리 3 — 만료/위변조 토큰 처리
**As a** 시스템
**I want to** 만료되었거나 위변조된 토큰을 가진 요청을 거부하고 싶다
**So that** 인증되지 않은 접근을 차단하고 사용자를 안전하게 재로그인시킬 수 있다

**Acceptance Criteria**
- [ ] 만료되었거나 위변조된 토큰으로 인증이 필요한 API를 호출하면 401 오류를 받는다.
- [ ] 클라이언트가 401 응답을 받으면 인증 상태를 초기화(로그아웃 처리)하고 로그인 페이지로 이동시킨다.

> **확인 필요:** 위 401 처리가 모든 API 요청에 공통으로 적용되는 전역 규칙인지, 아니면 특정 보호 API에만 한정되는지 PRD 7.3절 표현이 다소 일반적이다. 프론트 개발 시 axios 인터셉터 적용 범위를 명확히 확인해야 한다.

### 스토리 4 — 라우팅 가드
**As a** 시스템
**I want to** 인증되지 않은 사용자가 보호된 페이지에 접근하면 로그인 페이지로 보내고 싶다
**So that** 인증이 필요한 화면이 보호된다

**Acceptance Criteria**
- [ ] 보호된 라우트에 `isAuthenticated`가 false인 상태로 접근하면 `/login`으로 리다이렉트된다.
- [ ] 로그인 성공 후, 원래 접근하려던 경로(또는 정해진 기본 페이지)로 이동한다.

> **확인 필요:** "로그인 성공 후 이동할 기본 페이지" 경로가 PRD에 구체적으로 명시되어 있지 않다 (예: `/`, `/books` 등). 또한 이 "원래 경로로 복귀" 동작이 PRD 8절 수용 기준 체크리스트에는 별도 항목으로 명시되어 있지 않아, 7.4절 본문과 8절 체크리스트 간 정합성을 확인해야 한다.

### 스토리 5 — 로그아웃
**As a** 로그인한 사용자
**I want to** 로그아웃할 수 있다
**So that** 내 인증 정보를 더 이상 노출하지 않을 수 있다

**Acceptance Criteria**
- [ ] 로그아웃 동작 시 Zustand 스토어의 토큰/사용자 상태가 초기화된다.
- [ ] 로그아웃 후에는 보호된 라우트 접근 시 다시 로그인 페이지로 리다이렉트된다.

> **확인 필요:** 로그아웃을 트리거하는 UI(예: 헤더의 로그아웃 버튼 위치/문구)가 PRD에 명시되어 있지 않다. 본 PRD 범위가 "로그인 기능"으로 한정되어 있어, 로그아웃 버튼/메뉴 UI는 별도 확인이 필요하다.

## 3. 도메인 및 데이터 요구사항

### 3.1 인증 흐름 (개요)
1. 사용자가 로그인 폼에 userId, passwd를 입력한다.
2. 클라이언트가 로그인 API를 호출한다.
3. 서버는 `users` 테이블에서 사용자를 조회하고, bcrypt로 비밀번호를 검증한다.
4. 검증에 성공하면 서버가 JWT를 발급한다.
5. 클라이언트는 발급받은 accessToken을 Zustand persist 스토어에 저장한다.
6. 이후 인증이 필요한 요청에는 `Authorization: Bearer <token>` 헤더가 첨부된다.

### 3.2 JWT 정책
| 항목 | 값 |
|---|---|
| 알고리즘 | HS256 |
| 토큰 타입 | Access Token (Bearer) |
| 만료 시간 | 1시간(3600초), 환경변수로 조정 가능 |
| 시크릿 키 | 환경변수 `JWT_SECRET`(256bit 이상), 코드 하드코딩 금지 |
| Claims | 사용자 ID(`sub`), 이름(`names`), 권한(`roles`), 발급/만료 시각(`iat`/`exp`) |

### 3.3 비밀번호 암호화 정책
- 저장 시 `BCryptPasswordEncoder`로 해시 후 저장 (strength 10 권장).
- 검증 시 입력값과 저장된 해시를 `matches()`로 비교하며, 평문 비교는 금지된다.
- 평문 비밀번호는 로그, 응답, DB 어디에도 남기지 않는다.

### 3.4 토큰 저장 방식과 트레이드오프
- 본 기능에서는 Access Token만 발급하며, 클라이언트는 이를 Zustand `persist`(localStorage)에 저장한다.
- 이는 단순성을 위한 선택이며, localStorage 저장은 XSS 위험에 노출될 수 있다는 트레이드오프가 있다.
- 보안 강화가 필요할 경우 Access Token은 메모리에만 두고 Refresh Token을 httpOnly 쿠키로 운용하는 방식으로 확장 가능하다 (→ 9절 향후 확장).

### 3.5 데이터 모델 — `users` 테이블
| 컬럼 | 타입 | 제약 | 비고 |
|---|---|---|---|
| user_id | varchar(100) | PK, not null | 로그인 ID |
| passwd | varchar(200) | not null | bcrypt 해시 저장, 평문 금지 |
| names | varchar(50) | not null | 사용자 이름 |
| birth | varchar(10) | not null | |
| gender | varchar(10) | not null | |
| phone | varchar(20) | not null | |
| email | varchar(100) | not null, unique | |
| address | varchar(300) | not null | |
| addr_detail | varchar(300) | default '' | |
| postcode | varchar(10) | not null | |
| roles | varchar(10) | not null | 예: USER, ADMIN |
| create_at | timestamp | default null | |
| update_at | timestamp | default null | |

로그인 기능에서 실제로 사용하는 필드는 `user_id`(로그인 ID 식별), `passwd`(인증 검증), `names`/`roles`(인증 응답 및 JWT 클레임 구성)이다.

> 본 기획 단계에서는 회원가입 기능이 범위 밖이므로, `users` 테이블에 실제 데이터를 채우는 절차(시드 데이터 등)는 본 문서에서 다루지 않는다.

## 4. API 계약 안내

로그인 API(`POST /api/auth/login`), 인증 확인 API(`GET /api/auth/me`), 공통 에러 응답 포맷 등 구체적인 요청/응답 스키마는 `docs/specs/api-contract.md`에서 프론트/백 개발자가 별도로 확정한다. (`docs/login.md` 5절에 초안이 이미 작성되어 있으므로, 개발자는 해당 절을 그대로 검토·확정하는 방식으로 진행할 수 있다.)

## 5. 향후 확장 (Backlog)

`docs/login.md` 9절 기준.

- Refresh Token 도입 및 토큰 회전 (httpOnly 쿠키 방식)
- 로그인 실패 횟수 제한 / 계정 잠금
- 회원가입 기능 PRD 연동
- `roles` 기반 권한 분기 (ADMIN 등)
- `update_at` 기록 (마지막 로그인 시각 등 운영 정책에 따라)

## 6. 확인 필요 항목 (PM → 팀장 보고 대상)

추측하지 않고 명확화가 필요해 보류한 항목이다.

1. **401 인터셉터 적용 범위**: 클라이언트의 401 응답 처리(로그아웃 + 리다이렉트)가 모든 API 요청에 전역적으로 적용되는지, 특정 보호 API에 한정되는지 PRD 7.3절 표현만으로는 확정하기 어렵다.
2. **로그인 성공 후 이동할 "기본 페이지" 경로**: PRD 7.4절에 "원래 가려던 경로 또는 기본 페이지로 이동"이라고만 되어 있고, 기본 페이지의 실제 경로(`/`, `/books` 등)가 명시되어 있지 않다.
3. **"원래 경로로 복귀" 동작의 수용 기준 정합성**: PRD 7.4절 본문에는 있지만 8절 acceptance criteria 체크리스트에는 해당 항목이 명시적으로 없다. 의도적 누락인지 확인이 필요하다.
4. **`GET /api/auth/me` 호출 시점**: "앱 진입 시 토큰 유효성 확인 용도"라고 되어 있는데, 이것이 앱 최초 로드(새로고침) 시 1회 호출인지, 매 페이지 진입마다 호출인지 명시되어 있지 않다.
5. **로그아웃 트리거 UI**: 로그아웃 버튼/메뉴의 위치나 문구가 PRD에 명시되어 있지 않다. 본 PRD 범위가 로그인 기능으로 한정되어 있어, 별도 확인이 필요하다.

# 기능 명세 — 카테고리별 도서 리스트 화면 API 연동

> 본 문서는 `docs/bookList.md`(카테고리별 도서 리스트 API 초안)를 기반으로, 기획 관점에서 "무엇을 만들지(요구사항)"를 사용자 스토리 단위로 재구성한 것이다.
> 구체적인 API 엔드포인트/HTTP 메서드/요청·응답 JSON 스키마 등은 본 문서의 범위가 아니며, 다음 단계(프론트/백엔드 개발자 협의)에서 확정한다. `docs/bookList.md`의 스펙은 초안이며 아래 요구사항을 만족하는 범위 내에서 조정될 수 있다.

## A0. 참고한 기존 구현

`shop-front/src/components/BookListPage/BookListPage.jsx`, `shop-front/src/data/{itBooksData,novelData,selfDevData}.js`, `shop-front/src/pages/{ITBooks,Novel,SelfDev}/*.jsx`, `shop-back/src/main/java/com/booksshop/shopback/book/{Book,Category,Review}.java`, `shop-back/.../book/dto/*.java`, `shop-back/.../book/{BookController,BookService}.java`.

## A1. 배경 — 현재 상태

- 프론트: 3개 카테고리 페이지가 각각 22개짜리 mock 배열을 `BookListPage` 공통 컴포넌트에 넘긴다. `BookListPage`는 정렬(최신순/낮은가격순/높은가격순/리뷰많은순), 페이지네이션(10개/페이지), 베스트/신간 배지, 5칸 별점, 장바구니·바로구매 버튼(비로그인 시 alert)을 이미 클라이언트 로직으로 구현해 두었다.
- 백엔드: `Book`(카테고리 FK, title/subtitle/author/publisher/coverImage/listPrice/salePrice/publishedDate/bestYn/newYn 등)과 `Category`(types: IT/NOVEL/SELF) 엔티티, `Review`(rating: **1~5 정수**, 도서 상세 리뷰용) 엔티티가 존재한다. `BookController`/`BookService`에는 메인페이지 top-N 조회, 도서 상세 조회만 있고 **카테고리별 목록 조회 기능이 없다.**
- `docs/bookList.md` 초안: 엔드포인트 `/api/books/{types}`, 매개변수 page/size/orderType, 응답 필드 bookId/title/subtitle/author/publisher/publishedDate/totalReviewCnt/reviewRating/listPrice/salePrice/coverImage. 기본 정렬은 출판일 최신순(동률 시 제목순), orderType 값은 new/high/lower/reviewCnt.

## A2. 범위

### 포함
- 카테고리(IT/NOVEL/SELF)별 도서 목록을 실제 API로 조회하여 3개 페이지에 연동.
- 정렬 4종(최신순/낮은가격순/높은가격순/리뷰많은순), 페이지네이션(10개/페이지) 동작을 mock 대신 실제 데이터로 재현.
- 배지(베스트/신간), 평점, 리뷰 수, 정가/판매가/할인율, 표지 이미지 표시를 실제 데이터 필드로 재현.

### 제외 (이번 기능 범위 아님)
- 장바구니 담기 / 바로구매 실제 동작(현재 로그인 필요 alert만 있음) — 기존 동작 유지, 변경 없음.
- 리뷰 작성/조회 기능 자체(이미 도서 상세 페이지에 구현됨) — 이 기능은 그 결과값(평균 평점, 리뷰 수)을 "읽기"만 한다.
- 검색, 필터(가격대·출판사 등) — 문서 초안에 없으므로 범위 밖.
- API 엔드포인트/스키마 확정 — 다음 단계 몫.

## A3. 사용자 스토리 및 수용 기준

### US-1. 카테고리 메뉴 선택 시 실제 도서 목록 표시
사용자로서, 헤더 메뉴에서 IT서적/소설/자기계발서 중 하나를 선택하면 mock이 아닌 실제 DB에 저장된 해당 카테고리 도서 목록을 보고 싶다.

**수용 기준**
- `ITBooksPage`(IT), `NovelPage`(NOVEL), `SelfDevPage`(SELF)가 각각 자신의 카테고리 types 값으로 API를 호출하여 목록을 받아온다.
- 세 페이지 모두 기존과 동일하게 공통 컴포넌트 `BookListPage`를 통해 렌더링한다(페이지별 분기 로직 최소화).
- API 응답 도착 전(로딩 중)에는 로딩 상태를 사용자에게 표시한다.
- API 호출 실패 시 사용자에게 에러 상태를 표시한다(빈 화면으로 방치하지 않는다).
- 해당 카테고리에 도서가 0건이면 "표시할 도서가 없습니다" 류의 빈 상태를 표시한다(현재 mock은 항상 22건이라 이 케이스가 구현돼 있지 않음 — 신규 요구사항).

### US-2. 기본 정렬 — 출판일 최신순
사용자로서, 목록에 처음 진입했을 때 최신 도서부터 보고 싶다.

**수용 기준**
- 별도 정렬 조작 없이 진입 시 출판일 내림차순으로 정렬되어 표시된다.
- 출판일이 동일한 도서끼리는 제목 오름차순(가나다/사전 순)으로 2차 정렬된다.

### US-3. 정렬 옵션 변경
사용자로서, 최신순/낮은가격순/높은가격순/리뷰많은순 중 원하는 기준으로 목록을 다시 정렬하고 싶다.

**수용 기준**
- 기존 UI의 4개 정렬 버튼(최신순/낮은가격순/높은가격순/리뷰많은순)을 그대로 유지한다. ("인기순"은 이미 제거된 상태이며 이번 작업에서도 추가하지 않는다.)
- 가격 정렬은 판매가(salePrice) 기준이다.
- 리뷰많은순은 전체 리뷰 수(totalReviewCnt) 기준 내림차순이다.
- 정렬 기준을 변경하면 현재 페이지는 1페이지로 초기화된다(기존 동작 유지).

### US-4. 페이지네이션
사용자로서, 도서가 많을 때 한 번에 다 보지 않고 페이지 단위로 넘기며 보고 싶다.

**수용 기준**
- 한 페이지에 10건씩 표시한다(기존 PAGE_SIZE=10 유지).
- 처음/이전/다음/마지막 페이지 이동과 페이지 번호 직접 이동이 동작한다(기존 UI 유지).
- 총 도서 수(및 이를 통해 계산되는 총 페이지 수)는 실제 데이터 기준으로 정확해야 한다(현재 헤더의 "총 N권" 표시 포함).

### US-5. 베스트/신간 배지 표시
사용자로서, 어떤 책이 베스트셀러이거나 신간인지 배지로 구분해서 보고 싶다.

**수용 기준**
- `Book.bestYn == 'Y'`인 도서는 "베스트" 배지를, `Book.newYn == 'Y'`인 도서는 "신간" 배지를 표시한다.
- mock 데이터의 `coverColor`, 임의 `badge` 문자열 필드는 더 이상 사용하지 않는다.
- (동시에 Y인 경우의 처리는 "확인 필요" 섹션 Q2 참조.)

### US-6. 평균 평점 및 리뷰 수 표시
사용자로서, 각 도서의 평균 평점과 총 리뷰 수를 보고 구매 판단에 참고하고 싶다.

**수용 기준**
- 목록의 각 항목에 평균 평점(별점 아이콘 + 숫자)과 총 리뷰 수(totalReviewCnt)를 표시한다.
- **[사람 결정 — 2026-07-03] 평점 스케일은 10점 만점을 유지한다.** (PM이 제안한 "5점 만점 통일"안은 채택하지 않음.) `reviewRating`은 `Review.rating`(1~5 정수) 평균값에 2를 곱한 10점 만점 값으로 계산한다. 별점 컴포넌트(`BookListPage`의 `StarRating`)는 기존과 동일하게 `Math.round(rating/2)`로 5칸 중 채움 개수를 계산하는 로직을 그대로 유지한다.
- 리뷰가 0건인 도서는 평점을 표시하지 않거나 "리뷰 없음"으로 표시한다(별점 0개로 오인되지 않도록).

### US-7. 실제 표지 이미지 표시
사용자로서, 색상 placeholder가 아니라 실제 책 표지 이미지를 보고 싶다.

**수용 기준**
- 각 항목은 `Book.coverImage` URL의 이미지를 표시한다.
- `coverImage`가 비어있거나 로드 실패 시 대체 placeholder(예: 기존 색상 블록 또는 기본 이미지)를 표시한다(신규 요구사항 — 현재 mock엔 실패 케이스가 없음).

### US-8. 가격/할인율 표시 (변경 없음, 재확인)
사용자로서, 정가와 판매가, 할인율을 보고 싶다.

**수용 기준**
- 정가(listPrice), 판매가(salePrice), 할인율(= (listPrice - salePrice) / listPrice, 반올림 %) 표시 로직은 기존과 동일하게 유지한다. 계산식은 프론트에 이미 있으므로 백엔드가 별도로 할인율을 내려줄 필요는 없다(단, 이 판단은 다음 단계에서 재확인 가능).

### US-9. 비로그인 사용자 장바구니/바로구매 제한 (변경 없음)
사용자로서, 로그인하지 않은 상태에서 장바구니/바로구매를 누르면 안내를 받고 싶다.

**수용 기준**
- 기존 동작(비로그인 시 alert) 그대로 유지한다. 이번 기능은 이 부분을 변경하지 않는다.

### A4.2 정렬/페이지네이션 처리 위치 권고

- **서버 사이드로 이전할 것을 권고한다.** 현재는 mock 22건을 클라이언트에서 정렬·슬라이싱하지만, 실제 데이터는 카테고리별 도서 수가 늘어날 수 있고, `docs/bookList.md`에도 이미 `page`/`size`/`orderType` 서버 파라미터가 정의돼 있다. 또한 "출판일 동일 시 제목순" 같은 2차 정렬 규칙은 서버(SQL/JPQL ORDER BY)에서 처리하는 편이 프론트 재구현보다 안정적이다.
- 프론트는 정렬 버튼 클릭 시 서버에 orderType 파라미터를 다시 요청하고, 페이지 이동 시 page 파라미터로 재요청하는 방식으로 전환한다(현재의 "한 번에 다 받아서 클라이언트에서 정렬/슬라이스"하는 방식은 폐기).
- 총 건수(totalElements)는 서버 응답에 포함되어야 "총 N권" 표시와 총 페이지 수 계산이 정확해진다.

### A4.3 배지 판정 기준

- 베스트: `Book.bestYn == 'Y'`
- 신간: `Book.newYn == 'Y'`
- 우선순위/동시 표시 여부는 "확인 필요 Q2" 참조.

### A4.4 평점 스케일

- `Review.rating`은 1~5 정수(도서 상세 리뷰 작성 시 `StarRating` 컴포넌트가 5칸 기준으로 채워짐, `n <= rating`).
- **[사람 결정 — 2026-07-03] 카테고리 목록의 `reviewRating`(평균값)은 10점 만점을 유지한다.** (PM 제안 "5점 만점 통일"은 미채택.) 계산식: `Review.rating` 평균값 × 2. 목록 화면의 `StarRating`은 기존처럼 `Math.round(rating/2)`로 5칸을 채운다. 상세 페이지(`BookDetailPage`)의 5점 만점 표시는 이번 범위에서 변경하지 않는다 — 두 화면의 스케일이 다르게 유지되는 점을 인지하고 진행한다.

## A5. 확인 필요 (사람 확인 전 합리적 기본값 제시)

아래 항목은 문서/코드만으로 확정하기 애매하여, 기본값과 판단 근거를 제시하되 다음 단계(개발자 협의)에서 재확인이 필요하다.

**Q1. 평균 평점이 소수일 때 별점 UI를 어떻게 채울까?** — [사람 결정 — 2026-07-03: 10점 만점 유지로 확정, 아래는 확정된 처리 방식]
- 확정: `reviewRating`(10점 만점, 예: 9.8)을 `Math.round(rating/2)`로 5칸 중 채움 개수를 계산한다. 기존 `BookListPage`의 `StarRating` 로직을 그대로 유지(변경 없음).
- 근거: 부분 채움(반쪽 별) UI는 현재 CSS/컴포넌트에 없어 최소 변경으로 처리.

**Q2. 한 도서가 베스트이면서 동시에 신간이면 배지를 어떻게 표시할까?**
- 기본값 제안: 베스트 배지를 우선 1개만 표시한다.
- 근거: 현재 mock 데이터 구조가 단일 `badge` 필드(`'best'|'new'|null`)라 원래도 동시 표시를 가정하지 않았음. 베스트가 판매 실적 기반이라 신간보다 정보 가치가 크다고 판단.

**Q3. 리뷰 0건인 도서의 평점 표시는?**
- 기본값 제안: 별점 대신 "리뷰 없음" 텍스트를 표시하고 리뷰 수는 "0건"으로 표시한다.
- 근거: 별 0개로 표시하면 "평점이 낮다"로 오인될 수 있음.

**Q4. 출판일(`publishedDate`) 표시 포맷은?**
- 기본값 제안: `YYYY.MM.DD` (백엔드 `LocalDate`를 프론트에서 포맷팅) — 다만 목록 화면은 mock처럼 `YYYY.MM`(일자 생략)만 보여줄지도 선택 가능.
- 근거: 정렬 로직(최신순, 동률 시 제목순)이 일자 단위까지 비교해야 정확하므로 데이터 자체는 일자까지 유지하되, 화면 표시는 UX 선호에 따라 월까지만 노출할 수도 있음 — 디자인 협의 필요.

**Q5. 카테고리에 도서가 0건이거나 API 에러일 때의 화면 문구/디자인은?**
- 기본값 제안: 빈 상태 "등록된 도서가 없습니다", 에러 상태 "목록을 불러오지 못했습니다. 다시 시도해주세요." + 재시도 버튼.
- 근거: 현재 mock 기반 구현엔 이 케이스 자체가 없어 신규 정의가 필요. TanStack Query의 `isLoading`/`isError` 상태를 그대로 활용 가능(front-api-client 컨벤션).

**Q6. 정렬 변경/페이지 이동 시 URL 쿼리 파라미터에 상태를 반영할지(새로고침·뒤로가기 시 유지)?**
- 기본값 제안: 이번 범위에서는 반영하지 않는다(기존처럼 컴포넌트 로컬 상태로 유지).
- 근거: 문서 초안·기존 구현 어디에도 URL 동기화 요구사항이 없어 범위 확장으로 간주.
